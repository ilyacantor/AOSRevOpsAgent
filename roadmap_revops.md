# Pipeline Health Monitor - RevOps Roadmap

**Version:** 1.0  
**Last Updated:** November 2025  
**Project Status:** Demo-Ready Prototype with Backend DCL Framework

---

## Executive Summary

The Pipeline Health Monitor is a revenue operations analytics platform that provides pipeline health monitoring and CRM data quality insights. The system features a Data Connectivity Layer (DCL) backend architecture and a React frontend that can display data from integrated platform views or use demo data for testing.

**Core Value Proposition:**
- **Pipeline Visibility**: Dashboard showing opportunity metrics, risk scores, and health indicators
- **CRM Quality Insights**: BANT framework validation for opportunity data
- **Connector Testing**: Backend DCL with Salesforce, Supabase, and MongoDB connectors (viewable on Connectivity page)
- **Flexible Data Modes**: Platform integration (LIVE mode) or representative demo data (DEMO mode)
- **Modern Architecture**: React 19 + FastAPI with TypeScript and Tailwind CSS

**Architecture Note:** The Dashboard and Operations pages display data from platform views (when enabled) or mock data. The backend DCL connectors exist and are fully functional, accessible via the Connectivity page for connector health monitoring and drill-to-details exploration.

## Current Features (v1.0)

### ✅ Data Connectivity Layer (DCL)

**Status:** Backend Implementation Complete (Used by Connectivity Page)

The DCL backend serves as a connector management and testing layer:

- **Registry Pattern**: Dynamic connector registration and discovery
- **Standardized Interface**: All connectors expose `query(params)` with consistent response format
- **Health Monitoring**: Real-time connection health checks with 60-second caching via `/api/dcl/connectors` endpoint
- **Graceful Degradation**: Automatic fallback to mock data when sources are unavailable
- **Three Implemented Connectors**:
  - Salesforce (OAuth 2.0, SOQL queries) - configured via secrets
  - Supabase PostgreSQL (health scores table) - configured via secrets
  - MongoDB (user engagement analytics) - configured via secrets

**Current Usage:** The DCL backend is accessible via the Connectivity page, which shows connector health status and provides drill-to-details modals with up to 20 live records per connector. Dashboard and Operations pages currently use platform views or mock data, not direct DCL endpoints.

The dashboard provides pipeline analytics visualization:

**Interactive Features:**

**Operating Guide:**

### ✅ Connectivity Page - DCL Explorer

**Status:** Backend Framework Complete, Requires Credential Configuration for Live Testing

Connector status and drill-to-details functionality using backend DCL API:

**Connector Cards (via `/api/dcl/connectors`):**
- Health status endpoint (healthy/degraded/offline based on connection tests)
- Connection mode (LIVE if credentials exist in secrets, DEMO mock fallback otherwise)
- Data source metadata
- Health check caching (60-second TTL)

**Drill-to-Details Modal (via `/api/dcl/connectors/{name}`):**
1. **Overview**: Configuration, capabilities, description
2. **Health**: Health check results with error diagnostics
3. **Live Data**: Retrieves records from connector (requires valid credentials for live data)

**Supported Actions:**
- Click any connector to view detailed modal
- Force refresh health checks (bypass cache with `?force_check=true`)
- View sanitized configuration (secrets are hidden)

**Requirements for LIVE Mode (Unverified in Current Repo):**
- **Salesforce**: OAuth 2.0 secrets (SALESFORCE_INSTANCE_URL, SALESFORCE_CLIENT_ID, SALESFORCE_CLIENT_SECRET, SALESFORCE_REFRESH_TOKEN)
- **Supabase**: SUPABASE_URL and SUPABASE_KEY secrets
- **MongoDB**: MONGODB_URI and MONGODB_DATABASE secrets

**Important:** Backend DCL framework is implemented with mock fallbacks. Live connector record retrieval (up to 20 records per query) requires user-provided credentials and has not been fully validated with actual data sources in this repository. Verify credential configuration and test connector functionality in your environment before relying on live data.

**Status:** Framework Complete, Used Only by Connectivity Page

### ✅ Deployment Architecture

**Status:** Deployment-Ready, Tested on Replit

- **Development**: Vite dev server (port 5000) + FastAPI (port 8000) as separate workflows
- **Production**: FastAPI serves both API and static React build on one port

- **Run Command**: `uvicorn api:app --host 0.0.0.0 --port 5000`
- **Platform**: Replit Deployments tested, Render-compatible
- **Environment**: Replit Secrets for optional data source credentials

**Note:** Application runs successfully with or without live credentials (platform views or mock data fallback).

## Known Limitations

### 🔶 Architecture Reality Check

**Critical Note on Data Flow:**

The current architecture has **two separate data paths**:

1. **Dashboard & Operations Pages**: Use platform views (AosClient integration) when `VITE_USE_PLATFORM_VIEWS=true`, or mock data when disabled/unavailable. These pages **do not** call the backend DCL API endpoints directly.

2. **Connectivity Page**: Directly calls backend DCL endpoints (`/api/dcl/connectors`) to display connector health and live data from Salesforce, Supabase, and MongoDB.

**Implication:** The backend DCL connectors are fully implemented and functional, but they serve as a connector testing/monitoring interface (via Connectivity page) rather than the primary data source for analytics views. To unify the architecture in future versions, Dashboard and Operations could be refactored to query backend workflows that orchestrate DCL connectors, but this requires additional integration work.

### 🔶 Data Source Constraints

1. **Salesforce Rate Limits**
   - Subject to Salesforce API governor limits (varies by license)
   - No automatic retry/backoff for rate limit errors
   - Recommend implementing exponential backoff in future

2. **MongoDB Connection Pooling**
   - Single connection per request (no connection pool)
   - May hit connection limits under high concurrent load
   - Consider adding motor (async MongoDB driver)

3. **Supabase Row Limits**
   - Queries limited to 1000 rows by default
   - No automatic pagination for large health score tables
   - Recommend implementing cursor pagination

### 🔶 Functionality Gaps

1. **No User Authentication**
   - Currently single-tenant demo application
   - All users see same data
   - No role-based access control (RBAC)

2. **Limited Workflow Customization**
   - BANT validation rules are hardcoded
   - No UI for customizing thresholds (e.g., "stalled" = 30 days)
   - No admin panel for configuration

3. **Slack Integration**
   - One-way alerts only (no interactive buttons)
   - Single webhook (no multi-channel routing)
   - No alert scheduling or digest mode

4. **No Historical Tracking**
   - Dashboard shows current state only
   - No trend analysis over time
   - No opportunity stage history

5. **Limited Export Options**
   - No CSV/Excel export for reports
   - No PDF generation for executive summaries
   - Data only accessible via UI

### 🔶 Technical Debt

1. **Health Check Caching**
   - 60-second cache is in-memory (lost on restart)
   - No distributed cache (Redis) for multi-instance deployments
   - Cache invalidation is time-based only

2. **Error Logging**
   - Console logging only (no structured logging)
   - No centralized error tracking (e.g., Sentry)
   - Difficult to debug production issues

3. **Testing Coverage**
   - No automated test suite
   - Manual testing only
   - No CI/CD pipeline

4. **Frontend State Management**
   - Local component state only (useState)
   - No global state manager (Redux, Zustand)
   - Potential prop drilling as app grows

5. **API Versioning**
   - No API version strategy (/v1/, /v2/)
   - Breaking changes would affect existing clients
   - No deprecation policy
